# AI GitHub 热门项目推荐研究台设计文档

## 背景

目标是做一个以“发现 AI 相关 GitHub 高赞项目”为主的网页型研究助手。第一页重点不是做复杂管理后台，而是帮助用户快速回答两个问题：今天有哪些值得看的 AI 项目，以及为什么值得看。

本项目第一版优先支持手动刷新数据，后续再扩展自动更新。主题范围覆盖泛 AI，包括模型、Agent、RAG、语音、图像、训练框架及相关应用与基础设施。

## 产品目标

第一版产品需要同时满足以下目标：

1. 让用户快速发现值得关注的 AI GitHub 项目。
2. 推荐结果同时考虑历史影响力和近期热度，而不是只看累计 stars。
3. 在发现流程中自然承接研究动作，包括对比、标签归类、备注和导出。
4. 保持首版结构清晰、可扩展，避免一开始就做成过重的知识库系统。

不在第一版内的内容：

1. 自动定时更新。
2. 复杂账号体系和多人协作。
3. 高级个性化推荐算法。
4. 多数据源聚合。

## 核心定位

产品定位为“推荐研究台”。首页主任务是发现项目，研究动作围绕推荐流展开，而不是反过来让用户先进入管理视角。

推荐研究台的主路径为：

`发现 -> 快速判断 -> 加入研究 -> 对比/导出`

这个路径要求用户大部分时间停留在推荐榜单中，而不是频繁跳转页面。

## 页面结构

首页采用三栏结构，但三栏职责必须严格分离。

### 左侧：发现控制区

只负责“怎么找”，包括：

1. 关键词或主题。
2. 时间范围。
3. 语言或方向筛选。
4. 总星标与近期热度权重。
5. 手动刷新入口。

第一版保持轻筛选，不做复杂高级查询面板。

### 中间：推荐榜单主区

这是首页核心区域，承担绝大多数浏览行为。这里展示推荐项目卡片和榜单切换，如：

1. 综合推荐。
2. 本周热度。
3. 历史高赞。
4. 新上升项目。

每张项目卡片至少应包含：

1. 项目名称与简介。
2. Stars。
3. 最近活跃信息。
4. 语言与方向标签。
5. 推荐理由摘要。
6. 进入详情、加入对比、打标签、收藏等快捷动作。

卡片的目标不是完整展示所有信息，而是帮助用户快速判断“值不值得进一步研究”。

### 右侧：研究侧栏

只负责“研究动作”，不打断首页的发现流程。这里承接：

1. 当前选中项目的详情摘要。
2. 用户标签。
3. 备注。
4. 加入对比。
5. 导出研究清单。

右侧信息应该服务于当前浏览上下文，而不是变成第二个主页面。

## 推荐逻辑

默认榜单采用“综合推荐分”排序。这个分值不是只看总 stars，也不是只追短期热点，而是综合多个信号：

1. 总星标。
2. 近期活跃度。
3. 近一段时间热度变化。
4. 最近更新时间。
5. 主题匹配度。

第一版不要求算法复杂，但要求推荐理由可解释。每个项目卡片必须展示一条简短的排序理由，例如：

1. 总 stars 高，最近 30 天仍活跃。
2. 近期增长快，属于 Agent 热门方向。
3. 历史影响力强，且最近仍有维护。

解释性优先于绝对算法精度。

## 数据模型

第一版只保留四类核心对象，避免系统过重。

### 项目

用于存放标准化后的 GitHub 项目信息，包括：

1. 基础标识信息，如仓库名、链接、Owner。
2. 简介、语言、许可证、标签。
3. stars、forks、更新时间等基础指标。
4. 近期热度与活跃度相关指标。
5. 推荐理由与综合推荐分。

### 标签

用于用户归类和快速筛选，例如：

1. Agent。
2. RAG。
3. 训练框架。
4. 值得跟进。
5. 待深入研究。

### 研究记录

用于保存用户的轻量研究状态，包括：

1. 收藏。
2. 已读。
3. 屏蔽。
4. 备注。
5. 自定义标签关联。

### 对比列表

用于保存用户临时加入对比的 2 到 4 个项目。这个对象强调当前研究会话，而不是长期归档。

## 核心交互

首页内的核心动作保持短路径，尽量不跳页。

### 查看详情

点击项目卡片后，在右侧展开详情摘要或使用抽屉展示更多信息，而不是跳到单独详情页。这样用户可以持续停留在榜单浏览上下文中。

### 加入对比

用户可从榜单中将 2 到 4 个项目加入对比工作台。对比维度至少包括：

1. stars。
2. 最近活跃度。
3. 语言。
4. 许可证。
5. 方向标签。
6. 用户备注。

### 标签与备注

用户应能一键打标签，并以低打断方式补充备注。备注行为可以稍深，但不应让用户离开当前发现路径。

### 导出清单

用户在完成初步筛选和对比后，应能将当前研究结果导出为清单，作为后续阅读或跟进材料。

## 交互原则

第一版遵循以下原则：

1. 轻操作：收藏、打标签、加入对比应是一键动作。
2. 低打断：研究动作不应迫使用户频繁跳页。
3. 可解释：推荐结果必须说明原因。
4. 可裁剪：复杂能力以后再扩展，不压到首版。

## 技术结构

第一版采用轻量的前后端分离结构。

### 前端展示层

负责：

1. 推荐流展示。
2. 榜单切换。
3. 轻筛选。
4. 详情侧栏。
5. 对比区。
6. 标签、备注与导出交互。

### 数据抓取层

负责从 GitHub 拉取项目数据，并支持手动触发刷新。第一版以手动刷新为主，不引入复杂任务调度。

### 项目标准化层

负责把 GitHub 原始数据整理成统一项目结构，保证后续排序和展示逻辑稳定。

### 推荐排序层

负责基于基础指标和热度信号计算综合推荐分，并输出解释性推荐理由。

### 用户研究数据层

负责保存用户的收藏、标签、备注、屏蔽和对比列表等轻量研究数据。

系统被拆成这五个单元，是为了后续替换排序算法、增加 AI 子分类、接入自动更新时不影响前端主流程。

## 异常与边界处理

GitHub 相关数据存在天然边界，第一版必须允许“部分可用”。

### 数据侧异常

需要处理以下情况：

1. GitHub 接口限流。
2. 主题命中不准确。
3. 某些项目信息不完整。
4. 某些热度指标暂时缺失。

处理原则是保留旧数据并提示状态，而不是让整页失败。页面需展示数据更新时间，并在必要时提示“部分指标缺失”。

### 用户动作异常

收藏、标签、备注、对比和导出应彼此独立。某个动作失败时，只提示对应动作失败，不能影响推荐榜单继续使用。

## 测试重点

第一版测试聚焦在可靠性而不是功能堆叠。

### 排序稳定性

同一批项目输入时，综合推荐排序应可复现，推荐理由应与排序逻辑一致。

### 研究动作可靠性

加入对比、打标签、保存备注、导出清单等操作应正确反映到界面和数据结果中。

### 异常可用性

当 GitHub 返回不完整数据或手动刷新失败时，页面仍应保留可浏览的旧结果，并给出明确提示。

## 第一版范围总结

第一版交付重点如下：

1. 一个以发现 AI 热门 GitHub 项目为核心的网页界面。
2. 综合推荐榜单与基础榜单切换。
3. 轻量筛选。
4. 项目详情侧栏。
5. 标签、备注、收藏、屏蔽。
6. 2 到 4 项目对比。
7. 导出研究清单。
8. 手动刷新数据。

明确不包含：

1. 自动定时更新。
2. 复杂个性化推荐。
3. 多人协作。
4. 多平台数据聚合。

## 后续演进方向

在第一版稳定后，可按以下方向扩展：

1. 定时自动更新。
2. 更细颗粒的 AI 分类体系。
3. 更丰富的热度趋势分析。
4. 更强的导出格式。
5. 长期研究库能力。

## 设计结论

本项目第一版应优先做成“推荐研究台”，以推荐流驱动发现，以轻量研究动作承接用户的后续判断。页面结构、数据模型和技术拆分都围绕这一原则展开：先帮用户高效发现值得关注的 AI GitHub 项目，再让研究动作自然发生，而不是把首页做成过重的管理系统。
